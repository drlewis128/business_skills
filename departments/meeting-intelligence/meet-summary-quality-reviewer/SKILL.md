# Meeting Summary Quality Reviewer

This skill runs a structured quality review on every weekly meeting intelligence summary before it is sent to Matt Mathison, checking for BLUF presence, format compliance, PHI exposure, attribution accuracy, action item completeness, and word count. No summary goes to Matt without passing this review — a failed quality check returns the summary to the generation step with specific revision instructions.

## When to Use

- A weekly meeting intelligence summary has been generated and is ready for Dr. Lewis review before distribution
- An ad-hoc meeting summary is produced for Matt Mathison or an entity CEO and needs quality validation
- Dr. Lewis wants to audit summary quality across a set of prior outputs for program improvement
- A summary fails quality check and the revision workflow needs to be initiated

## Summary Quality Reviewer Framework

```

=====================================================

SUMMARY METADATA

  - Review date: [date]

  Run each check; mark PASS / FAIL / WARNING

      - Does the BLUF give Matt enough to understand what happened and what he needs to do?

      - Total words in Matt's summary section: [N]
      - Target: ≤200 words
      Result: PASS (≤200) / FAIL (>[200]) — [actual count]
      Revision instruction if FAIL: [which sections to compress]

  [ ] PHI CHECK
      - Any individually identifiable health information? (member names; SSNs; diagnoses; claims)
      - Any reference to Allevio member-level data?
      Result: CLEAR / PHI DETECTED — [exact location and content]
      Action if DETECTED: HALT; notify Dr. Lewis; do not distribute

  [ ] CONFIDENTIAL DATA CHECK
      - LP names or capital amounts (HIVE)?
      - Unapproved client names (Column6)?
      - Deal counterparty names without Dr. Lewis approval?
      - Employee PII?
      Result: CLEAR / FLAGGED — [type and location]

  [ ] ACTION ITEMS COMPLETE

      - No action items assigned to "the team" or "leadership" without a named individual
      - Vague actions flagged: [list any that fail specificity test]
      Result: PASS / FAIL — [N items failed specificity; list]

  [ ] DECISIONS CAPTURED
      - Every decision mentioned in source meetings appears in the summary?
      - Each decision has: what was decided; who decided; date
      - No decisions are implied without being explicitly stated
      Result: PASS / FAIL — [missing decisions if any]

  [ ] ATTRIBUTION ACCURACY
      - Are commitments and statements attributed to the correct person?
      - Does any action item attribute a commitment to someone who didn't make it?
      - Flag any attribution that cannot be verified from transcript
      Result: PASS / FAIL / UNCERTAIN — [issues if any]

  [ ] FORMAT COMPLIANCE
      - Follows the required template structure for this summary type?
      - Section headers present and in correct order?
      - Table formatting correct (if tables used)?
      Result: PASS / FAIL — [format deviations]

  [ ] TONE CHECK
      - Is the summary direct with no hedging language ("it seems," "perhaps," "might")?
      - No filler phrases or corporate speak?
      - Matt's items are called out explicitly, not buried?
      Result: PASS / FAIL — [examples of hedging if found]

  - Non-critical failures: [N]

REVISION INSTRUCTIONS (if HOLD)

  1. [Exact revision instruction with location in document]
  2. [Exact revision instruction]
  3. [...]
```

## Output Contract

- **Any PHI detection is an automatic HOLD with no exceptions.** When the PHI check finds any individually identifiable health information — regardless of how minor or incidental the reference — the entire summary is held, the specific PHI location is documented, Dr. Lewis is notified immediately, and a HIPAA breach assessment is initiated before any revision or distribution occurs. This is not a fixable formatting error; it is a compliance event that must be handled before the summary workflow resumes. A revised summary after PHI removal requires a full re-review, not an expedited clearance.

- **Action item specificity is enforced at the individual level, not the team level.** "Operations team to follow up" fails the action item check every time. An action item with no named individual owner cannot be tracked, cannot be escalated when overdue, and cannot feed the closure rate metric that Matt Mathison uses to assess organizational accountability. The quality reviewer identifies every vague or team-attributed action item and returns them with revision instructions that name the specific individual who must own the item, based on the source transcript if necessary.

- **The 200-word limit for Matt's summary section is a respect-for-attention constraint, not a style preference.** Matt Mathison runs MBL Partners and its portfolio companies — a weekly summary that requires more than 200 words to convey the essential points has failed to do the synthesis work that meeting intelligence exists to do. When a summary exceeds the word limit, the quality reviewer provides specific compression instructions identifying which sentences convey information already captured elsewhere, which context is available on request rather than required in the brief, and which language can be tightened without losing meaning.

## System Dependencies

### Reads from
- Generated meeting intelligence summaries (from weekly summary builder or entity brief skills)
- Source transcript excerpts (for attribution verification)
- Prior approved summaries (for format reference comparison)

### Writes to
- Quality review log: SharePoint /Meeting Intelligence/Quality-Reviews/[date]-review-[summary-id].md
- Revision request (returned to summary generation workflow when HOLD issued)
- Distribution authorization record (when APPROVED)

### HITL Required
- Dr. Lewis: all quality reviews are performed by or under the direct supervision of Dr. Lewis; the skill supports the review, not replaces it
- Matt Mathison: has no role in the quality review process; receives only approved summaries

## Test Cases

1. **Golden path:** Weekly MBL summary submitted — BLUF present, 187 words, no PHI, all action items have named owners and due dates, 3 decisions captured with attribution, format correct, no hedging — 9/9 checks pass, APPROVED FOR DISTRIBUTION.
2. **Edge case:** A summary is 220 words — only 20 over limit. Quality reviewer must not approve it with a warning; it must HOLD with specific compression instructions that reduce it to ≤200 without losing any substantive content. The revision instructions identify the exact sentences or phrases to compress, not just flag "too long."
3. **Adversarial:** An action item reads "Dr. Lewis to follow up with Allevio on compliance question" — attributed correctly to Dr. Lewis with no due date. Attribution passes (Dr. Lewis said it in the meeting), but action item fails the completeness check for missing due date. Reviewer must flag the due date gap and return for revision, while correctly noting the attribution itself is accurate.

## Audit Log

Every quality review is stored in SharePoint with the summary ID, review date, checklist results, overall score, distribution authorization status, and (if applicable) revision instructions and re-review timestamp. The quality review log is used monthly to identify recurring failure patterns (e.g., consistent action item specificity failures) that indicate a systematic issue in the summary generation process rather than one-off errors. Dr. Lewis reviews the quality trend monthly as part of the program ops review.

## Deprecation

This skill is deprecated when the meeting intelligence summary generation pipeline achieves sufficient quality through automated validation that a separate quality review step is no longer necessary — specifically, when the automatic PHI check, action item completeness check, and word count enforcement are embedded directly in the generation workflow with the same rigor this skill provides. Until then, human-in-the-loop review via this skill is required for every summary. Dr. Lewis approves deprecation.
